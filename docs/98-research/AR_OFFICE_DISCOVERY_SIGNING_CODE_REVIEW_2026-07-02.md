# A&R Office — Artist Scouting, Discovery & Signing — Code Review
**Date**: July 2, 2026 · **Reviewer**: Claude (read-only session) · **Branch reviewed**: `claude/onboarding-jira33`
**Scope**: The A&R Office feature end to end — start a sourcing operation (Active / Passive / Specialized) → weekly discovery of an artist → display discovered artists → sign one to the roster. Covers `client/src/components/ar-office/*`, `arOfficeMachine.ts`, `gameStore` A&R/sign actions, `server/routes/arOffice.ts`, the sign endpoint in `server/routes/artists.ts`, and `processAROfficeWeekly` in the shared engine. High-severity findings were verified directly in source; breadth findings came from two mapping passes.
**Cross-reference**: Ownership/IDOR gaps in these routes are catalogued in `OWNERSHIP_AUTHORIZATION_SWEEP_2026-07-02.md` (arOffice ×4, sign-artist) and are **not** re-listed here except where they compound a correctness bug (B1).

## Background / how the feature works
- **Client**: `/ar-office` route → `pages/AROffice.tsx` → `components/ar-office/AROffice.tsx` (orchestrator) using `useMachine(arOfficeMachine)`. Sourcing mode is chosen in `SourcingModeSelector`; Specialized opens `GenreSelectionModal`. Discovered artists render in `ArtistDiscoveryTable`; signing calls `gameStore.signArtist`.
- **Server**: `POST/ar-office/start` reserves a focus slot and writes `arOfficeSlotUsed` + sourcing params; `advanceWeek` → `processAROfficeWeekly` picks exactly **one** artist and appends it to `game_states.flags.ar_office_discovered_artists`; `GET /ar-office/artists` returns the discovered list; `POST /api/game/:gameId/artists` signs.
- **Engine** (`game-engine.ts:550-825`): completion is gated solely on `arOfficeSlotUsed` (no elapsed-week check), so the op completes on the *next* advance. Pool = `data/artists.json` (**only 6 artists**, `art_1`–`art_6`); discovery excludes signed + already-discovered **by lowercased name** (JSON ids like `art_4` ≠ DB UUIDs). Sourcing: `active`/`specialized` = argmax(`talent+popularity`) (specialized filters by genre first), `passive` = seeded random.
- **Persistence**: schema columns `arOfficeSlotUsed / arOfficeSourcingType / arOfficePrimaryGenre / arOfficeSecondaryGenre / arOfficeOperationStart` (`shared/schema.ts:275-279`); everything else lives in the `flags` JSONB blob.

## Bugs (confirmed, ranked)

**B1 — Sign-artist fully trusts the client for cost and stats.** `POST /api/game/:gameId/artists` reads `signingCost = req.body.signingCost || 0` (`artists.ts:168`) and parses the *entire* artist record from `req.body` via `insertArtistSchema.parse({...req.body})` (`:189`) — talent, popularity, archetype, `weeklyCost`. The server **never** cross-references the discovered-artist record in `flags.ar_office_discovered_artists`, and `data/artists.json` carries authoritative `signingCost`/`weeklyCost`/stats that are ignored. `gameStore.signArtist` posts the client's `artistData` verbatim (`gameStore.ts:850-854`). A crafted request signs a max-stat artist for `$0` (or below the real cost). Compounds the ownership gap (no `userId` check on this route) — the two together mean *anyone* can add a fabricated artist to *any* game for free.

**B2 — Signing is non-atomic despite the "transaction-like" comment.** `createArtist` (`:196`), the money deduction `updateGameState` (`:200`), and the flags `updateGameState` (`:245`) are three separate awaits with no DB transaction. A crash between them leaves an artist with no payment (or money gone with stale flags). Both `updateGameState` calls also derive from the `gameState` snapshot read at `:171`, so a concurrent write to `flags` is lost.

**B3 — `GET /ar-office/artists` mutates state and uses unseeded `Math.random()`.** In the legacy fallback branch (`arOffice.ts:344-356`), a read request picks a random artist via `Math.random()` and **writes** it back with `updateGameState`. A GET with side effects, non-deterministic (breaks the seeded-RNG/save-reproducibility guarantee), and it filters the pool on the JSON `!a.signed` field (`:348`) — inconsistent with the name-based filtering used everywhere else in the feature.

**B4 — `signingCost` fallback defaults disagree between deduction and affordability.** The store deducts `artistData.signingCost || 0` (`gameStore.ts:858`), while `ArtistDiscoveryTable` gates affordability and displays cost with `artist.signingCost || 5000` (`ArtistDiscoveryTable.tsx:279,292,299`). If `signingCost` is ever undefined, the button blocks at $5,000 but the store would deduct $0 — the UI and the economy disagree.

**B5 — The 3-artist roster cap is UI-only and inconsistent.** The cap is enforced solely as a disabled button in `ArtistDiscoveryTable.tsx:293,300` (`signedArtists.length >= 3` → "Roster Full"). It is **not** enforced server-side and **not** present in the parallel `ArtistDiscoveryModal` sign path (see E1). Signing via the modal, or directly against the API, bypasses the cap entirely — so either the cap is a real rule that's unenforced, or it's dead UI. Duplicate-by-name is the only server-side signing guard (`artists.ts:182-186`).

**B6 — `signed_artists_count` is dead and off-by-one.** Written at `artists.ts:208` as `currentArtists.length + 1`, but `currentArtists` is fetched *after* `createArtist` (`:206`) so it already includes the new artist — the `+1` double-counts. The value is **never read anywhere** (verified by grep across client/server/shared), and its comment ("for GameEngine weekly costs") is false; weekly artist salaries are computed from the actual roster, not this flag.

**B7 — Active and Specialized sourcing are deterministic argmax; only Passive is random.** `selectBestArtist` returns the single highest `talent+popularity` artist (`game-engine.ts:638-658`); `active` and `specialized` both use it (specialized after a genre filter), `passive` uses seeded random (`:700`). So repeated Active scouting surfaces the remaining artists in strict descending quality order with zero variance, and — since Active always yields the best remaining — Passive and Specialized are strictly dominated except when genre-targeting. There is no quality-tier system, no probabilistic outcome, and no config; "three sourcing modes with different risk/reward" is not what the code does.

**B8 — Client flag-fallback reads the singular legacy keys, not the canonical array.** The engine writes both `ar_office_discovered_artists` (plural, canonical) and legacy `ar_office_discovered_artist_id`/`_info` (single). The store's flag fallback in `loadDiscoveredArtists` reads only the **singular** legacy keys (`gameStore.ts:1134-1135`); the plural array is never read client-side except via the server's enriched GET response. If the GET returns empty but flags hold a multi-artist array, the client surfaces at most one (legacy) artist.

**Verified NOT a bug (noted to prevent a false positive):** `pending_signing_fees` → `processPendingSigningFees` (`game-engine.ts:403-450`) only adds an `expense_tracking` line to the week summary; the money was already deducted at signing time. It is report-only, not a double-deduction (same pattern as project costs).

## Efficiency / smart-minimal-code findings
- **E1 — Two parallel discovery+sign UIs.** `ArtistDiscoveryModal.tsx` (mounted by `ArtistRoster.tsx:341` and `ArtistsLandingPage.tsx:393`) and `ar-office/ArtistDiscoveryTable.tsx` (mounted by the A&R page) both hit `GET /ar-office/artists` + the sign flow, with duplicated archetype helpers, search/filter, and sign handlers — and they filter signed artists differently (modal by id-or-name, table by name-only). One should be the single source.
- **E2 — Dead XState machine artifacts.** `COMPLETE_SOURCING` handler (never dispatched), named guards `hasFocusSlots`/`isValidSourcingType` (logic is inlined instead), and all four `notify*` callbacks (the `useMachine` input passes only `{gameId}`, so they're permanently `undefined`). The machine's own tests exercise mostly this dead logic.
- **E3 — Unused service module.** `services/arOfficeService.ts` exports `getAROfficeStatus`, `syncAROfficeSlots`, `validateSlotAvailability` — none imported; the store reimplements them inline and even collides on the name `getAROfficeStatus`.
- **E4 — ~130 lines of legacy dual-format handling** in `GET /ar-office/artists`: singular-vs-plural flag formats, an in-GET migration write (`arOffice.ts:180-198`), and layered fallbacks. Collapsing to the array format alone would remove most of the endpoint.
- **E5 — Signed-artist name filtering implemented in five places**, all name-based: engine (`625-628`), GET (`226-228`), sign (`182-186`), plus two client filters. Should be one shared helper; the underlying fragility is that identity keys on **name** because content ids ≠ DB UUIDs.
- **E6 — Dead / misleading operation-lifecycle state.** `flags.ar_office_start_week` is written "for completion validation" (`arOffice.ts:42`) but never read — completion never checks elapsed weeks. `arOfficeOperationStart = Date.now()` (`:50`) is wall-clock (non-deterministic, breaks save reproducibility), unused for gating, and left stale by cancel (`:88-95` never nulls it).
- **E7 — Debug noise in production paths.** `processAROfficeWeekly` logs `[A&R DEBUG]` throughout and dumps the entire flags object as JSON on every completion (`game-engine.ts:787`); the sign and GET handlers are similarly verbose.
- **E8 — Scattered hardcoded values.** Roster cap `3`, signing `5000`/weekly `800`/age `25` fallbacks, the 11-genre list in `GenreSelectionModal`, and default focus-slots `3` in several files — none in balance config. Also `process.env.NODE_ENV` used in a Vite client component (`ArtistDiscoveryTable.tsx:106,145`) instead of `import.meta.env.DEV`; and a dead local `availableSlots` (`gameStore.ts:526`) plus a stale TODO for already-implemented genre params (`ar-office/AROffice.tsx:310`).

## Wiring gaps
- **W1 — Focus-slot double-count window.** The machine gate is `(focusSlotsUsed + reservedLocally) < focusSlotsTotal`; `START_SOURCING` bumps `reservedLocally` while `consumeAROfficeSlot` bumps `gameState.usedFocusSlots`, which flows back as `focusSlotsUsed` via `SYNC_SLOTS` — transiently counting the same slot twice until completion/cancel runs `releaseLocalSlot`. `FocusSlotStatus` reads the machine context while the rest of the UI reads `gameState.usedFocusSlots`, so the header bar and the metrics dashboard can disagree during the lag.
- **W2 — Cancel can desync client vs server.** `cancelAROfficeOperation` swallows the API error but always calls `releaseAROfficeSlot()` in `finally` (`gameStore.ts:1099-1101`) — the client frees the slot even when the server cancel failed, leaving the server still "busy."
- **W3 — Signing costs no creative capital.** Projects and release planning each consume 1 creative capital; signing consumes only money. If intended, fine; flag because it's an asymmetry in the resource economy that isn't documented.
- **W4 — Zero server/engine test coverage.** No test exercises `processAROfficeWeekly`, the four arOffice routes, sourcing-mode selection, genre filtering, discovery→persist→sign→removal, or signing cost/validation. Only `tests/features/ar-office-machine.test.ts` runs — and it tests the (largely dead) client machine transitions. The route-manifest snapshot asserts registration only. This is the same gap that let the ownership and recording-session bugs ship unnoticed.
