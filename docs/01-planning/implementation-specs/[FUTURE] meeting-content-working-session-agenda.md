## (i) Why this session is the gating item

Two knob-tuning passes (#168, #169) plus the delegation arc itself (#167) all landed clean — golden-master byte-identical or audited, suites green, mechanics verified live. The playtest verdict on the **machinery** is consistently positive: three-lane clarity "works" (with one blur noted below), the digest "works," escalation "works" mechanically. But every qualitative comment this round converges on the same root cause, in the designer's own words:

> *"As we know - we need to rework the meetings and choices. The system is all wired correctly but a lot of the situations still feel random."* — round 3, `auto_vs_neglect_legibility`
> *"Not random but definitely not high stakes"* — same section, second answer
> *"This will land better when we rework the meetings."* — round 3, `disloyal_in_character` (both answers)
> *"Not based on current meetings and outcomes."* — round 3, `escalation_crisis_legibility`

The delegation arc's autonomous-resolution and escalation systems **amplify** whatever the underlying meeting pool already is — a disloyal exec's self-serving pick, an escalation crisis, a "made without you" digest line, all read straight off the authored `choices[]` for that meeting. Tuning knobs (#168/#169) cannot fix a content problem; every further mechanical pass on top of low-stakes content just makes low-stakes content fire more often, on a schedule, without the player's hand on it. **Content is now the bottleneck, not mechanics.** This session is the next arc.

## (ii) Agenda items A–E

### A. Meeting choices lack stakes (designer's #1 finding; root of B/C/E)

- **What the playtest showed:** round 3 `auto_vs_neglect_legibility.exposure = ["couldnt_tell"]`, `feel = "flat"`; both freeform answers name "rework the meetings and choices" directly, twice independently distinguishing "random" from the real complaint — "not random but definitely not high stakes." The mechanism (loyalty bands, mood risk-appetite, AUTO-vs-neglect divergence) is correctly wired and legible in isolation (round 3 confirms the digest itself "works," `read_it_every_week`), but the *choices being routed through it* don't carry enough weight to make the routing feel consequential.
- **What exists already:** the full three-band choice-policy machinery (`[READY] executive-delegation-and-trust-plan.md` §3.1/§4.3) is live and correctly reading whatever `choices[]` an exec's meeting authors. `/admin/actions-viewer` (`client/src/admin/ActionsViewer.tsx`, route `/admin/actions-viewer`) already supports full meeting/choice editing with a live balance-knob context strip and dominance-lint gating at save — the tool for this session already exists, nothing to build first.
- **What the session must produce:** a working definition of "stakes" for this game's choices (money magnitude alone is not it — the arc's autonomous spend already routes real, sometimes large, money and still reads flat) and a first editing pass over a representative slice of the 74 choices (start with the four core-four archetype pools, since B/C depend on them) raising the felt weight of at least the meetings exercised in this round's playtest (`ar_discovery`, `cmo_scandal`, `cco_creative_clash`, the `action_1760807005433` "Employee Effectiveness" CCO meeting, distribution meetings touched by Pat's disloyal reads).
- **Prep the orchestrator can do beforehand:** a per-meeting stakes audit — for every meeting in `data/actions.json`, tabulate choice-effect magnitude spread (is there a real trilemma or do choices cluster near-identical?), whether `self_serving_hint` is set where the disloyal-band scorer's argmax is weak (cross-reference C91's three flagged meetings), and whether the meeting's `requires`/`reactive_trigger` gives it situational grounding or it's a generic always-eligible pick. Deliver as a sortable table so the session opens on the worst offenders, not a blind scroll through 74 entries.

### B. CEO pool needs a "rarer and heavier" authoring pass (+ CEO/exec lane blur)

- **What the playtest showed:** round 3 `three_lane_clarity.exposure = ["ceo_and_exec_blurred"]`; the two designer answers split — "No" then "Yes" — suggesting the lane distinction is understood in principle but not always legible meeting-to-meeting, which tracks with a CEO pool that today is mechanically identical in *feel* to the exec pool (3 regular + 1 reactive, same authoring density) despite carrying different lapse semantics (genuinely lapses vs. autonomously resolves).
- **What exists already:** the plan doc already specced this exact gap and deliberately left it structural-only (`[READY] executive-delegation-and-trust-plan.md` §2 "Immediate structural consequences," §8 point 3, §13 "Deferred to a live content session"). The `requires`-gating mechanism used for meeting relevance already exists and is authoring-ready — CEO meetings can be gated to fire only on real strategic inflection points, exactly as exec meetings are gated to real label state.
- **What the session must produce:** a smaller, `requires`-gated CEO pool where each meeting is a genuine strategic fork with real forfeit-cost if lapsed (the CEO lane's lapse semantics are "the moment passed, no effect" — that only lands if the moment was worth having). This is also the fix for the blur: if CEO meetings become visibly rarer and heavier while exec meetings stay frequent and delegable, the two lanes should differentiate themselves without needing UI changes.
- **Prep the orchestrator can do beforehand:** enumerate the current CEO pool (3 regular + `ceo_chart_debut_strategy`'s reactive slot) with their current `requires` tags (if any) and eligibility frequency, so the session starts from "here's what fires today and how often" rather than a cold read.

### C. Disloyal self-serving picks read as in-character only after the meeting rework

- **What the playtest showed:** round 3 `disloyal_in_character.exposure = ["pat_distribution"]`, `feel = "flat"`, both answers: "This will land better when we rework the meetings." The archetype-scoring heuristic itself (§4.3.1 of the plan — Mac chases risky pet signings, Sam overspends on campaigns, Dante indulges experimental creative, Pat caps upside conservatively) is not in question; the designer is saying the *choices being scored* don't carry enough archetype flavor for the argmax to read as a character beat rather than a mechanical pick.
- **What exists already:** the `self_serving_hint: true` escape hatch (`data/actions.json`, currently set on exactly 2 of 74 choices: `cmo_scandal.spin_collaboration`, `ar_genre_shift.chase_trends`) already overrides the numeric-proxy scorer when the archetype's narratively-correct pick disagrees with the argmax — this is C91 in the debt backlog, logged the same day the arc shipped, with three meetings already flagged as weak-archetype under the numeric-only heuristic: `ar_discovery` (head_ar), `cco_creative_clash` (cco), and `action_1760807005433` ("Employee Effectiveness," cco — note the non-standard timestamp id, called out in the `[REFERENCE]` doc's known-discrepancy section).
- **What the session must produce:** the C91 sweep — walk every exec's pool against the Character Bible archetype table (plan §4.3.1) and set `self_serving_hint: true` wherever the mechanical argmax and the narratively-correct pick diverge. This rides along with the A pass (raising stakes on the same choices makes the archetype-driven pick more visible when it fires) and should be sequenced together, not as a separate editing pass over the same file.
- **Prep the orchestrator can do beforehand:** for each exec's meeting pool, run the disloyal-band scorer (`shared/engine/executiveAutonomy.ts` `scoreSelfServing`) offline against every meeting and report the argmax choice + its score margin over the runner-up; low-margin or narratively-implausible argmaxes are the `self_serving_hint` candidates. This turns "review 74 choices" into "review the dozen the scorer is uncertain about."

### D. C92 — authored past-tense outcome summaries per choice

- **What the playtest showed:** live-notes Entry 1 (2026-07-12) — the player read the digest's raw choice `label` ("Accept their terms, worth the risk") and couldn't tell if it was reporting what the exec did or instructing the player what to do. The render-site reframe (prefix "Their call: …", header "Made Without You") shipped same-session and resolves the immediate confusion, but the underlying label is still doing double duty as both the pre-decision prompt (second-person, imperative) and the post-hoc summary (should be past-tense, third-person, report-of-fact).
- **What exists already:** nothing yet — this is the one item here that needs a schema change, not just content. C92 (technical-debt-backlog.md) specs it: an additive, optional `outcome_summary?` field on the choice schema (`shared/api/contracts.ts`), read by the WeekSummary/results renderers with fallback to `label` when absent — fully backward-compatible, no migration.
- **What the session must produce:** the schema addition (small, mechanical — can be a short slice ahead of or alongside the content pass) plus authored past-tense summaries across the 74 choices as content is otherwise touched for A/B/C. Recommend NOT a separate full pass over all 74 — write `outcome_summary` for every choice touched during the A/B/C stakes-and-archetype editing anyway, so the two passes amortize into one editing session per meeting rather than three.
- **Prep the orchestrator can do beforehand:** draft the Zod schema diff + the WeekSummary/results read-site fallback logic ahead of the session so the mechanical plumbing is ready the moment authored copy starts landing — the content pass shouldn't be blocked on a schema PR mid-session.

### E. Escalation works mechanically but prose→meeting linkage is thin

- **What the playtest showed:** round 3 `escalation_crisis_legibility.feel = "works"`, `exposure = ["crisis_after_ignore"]`, but the two designer answers diverge sharply: "Felt connected." then "Not based on current meetings and outcomes." Read together: the escalation *trigger* (ignore an urgent meeting at low loyalty → crisis next week, plan §5) is legible as a mechanic, but the escalation *event's content* doesn't visibly trace back to which meeting was ignored or what the exec would have done.
- **What exists already:** the injection mechanism is fully live (existing #162 mandatory-side-events pipeline, `flags.pending_side_event`, one focus slot consumed) with four escalation-only stub events already wired and excluded from the weekly weighted roll (`data/events.json`: `escalation_ar_botched_signing`, `escalation_cmo_narrative_lost`, `escalation_cco_artist_walkout`, `escalation_dist_deal_collapsed` — one per core-four archetype). These are explicitly **placeholder/stub prose, sufficient for tests only** (plan §5.3, §13) — this was always the deferred half of Tier 2.
- **What the session must produce:** authored prose for the four escalation events that references the *specific kind* of urgent meeting that was ignored (not a generic "a crisis happened" — the archetype-specific fallout: a botched signing because Mac's pet-signing instinct went unchecked, a narrative lost because Sam's campaign call went unmade, etc.), so the crisis reads as the direct, in-character consequence of the escalation trigger rather than an unrelated bad-luck event landing the same week.
- **Prep the orchestrator can do beforehand:** pull the current stub text for all four escalation events plus the urgent/reactive meeting pool each archetype could plausibly have ignored to trigger them, so the session can write the causal link directly rather than reconstruct it from the engine.

## (iii) Design decisions to make (can be same session or async)

These are product/economy calls, not content-authoring passes — they gate content decisions above (a flop threshold decision shapes how "stakes" get authored in A; a CC-economy decision shapes what CCO meetings beyond "Employee Effectiveness" should grant). Can be decided async ahead of the session so the content pass isn't blocked mid-discussion.

### F. Flop trigger is practically unreachable in competent play

**The finding (PR #169 description, analysis rider):** the flop trigger — `totalInvestment >= flop_investment_floor` (default $10,000) AND `releaseWeekRevenue < flop_revenue_ratio × totalInvestment` (default ratio 0.10, i.e. release-week revenue under 10% of investment) — is gated in `shared/engine/processors/ReleaseProcessor.ts:1574-1641`, config in `data/balance/progression.json` (`reputation_system.flop_penalty`, `.flop_revenue_ratio`, `.flop_investment_floor`, plus slice-adjacent `.flop_artist_mood_penalty`). Even a weak, unpromoted artist tends to out-earn the 10%-of-investment bar in release week, so the bar is rarely crossed in competent play. Round 2's own form shows `flop_drama.exposure = ["never"]` — the mechanism has never been observed live despite #169 sharpening its penalties (mood −8→−12, reputation 5→8); sharpening penalties that never trigger tunes a knob nobody turns.

**Options (recommendation marked, not decided):**
1. **Raise `flop_revenue_ratio`** (e.g. 0.10 → 0.25–0.35) so a mediocre release-week performance — not just a catastrophic one — can cross the bar. Cheapest change (one config value), directly attacks the root cause (the bar is too low relative to typical release-week revenue), fully config-driven — no code change, GM-safe (same code path, new threshold).
2. **Lower `flop_investment_floor`** (e.g. $10,000 → $5,000–$7,000) so more releases are even eligible for the flop check at all. Addresses a different failure mode (maybe most releases never clear the investment floor in the first place) but doesn't fix the ratio problem if a release *does* clear the floor and still out-earns 10% comfortably.
3. **Make flop probabilistic near the bar** (a release within some band of the ratio — e.g. 10–25% of investment — has a *chance* of triggering the flop penalty, scaling down to certainty below the floor) rather than a hard boolean cliff. Most design work (new RNG draw, needs isolated-seed discipline per the arc's Ground Rule 4, GM blast radius), but softens the all-or-nothing cliff and could make a "near-miss" release feel tense rather than safely irrelevant.

**Recommendation:** option 1 (raise `flop_revenue_ratio`) as the cheapest, most surgical fix — it is a pure config change against a mechanism that is otherwise fully correct and already wired end-to-end (penalty application, mood hit, WeekSummary `flop` entry, reputation math). Options 2/3 are worth keeping on the table if a config-only ratio change turns out insufficient once actually exercised in play (which round 2's `exposure: ["never"]` means nobody has yet confirmed either way).

### G. Creative Capital has exactly one positive source in the game

**The finding:** grepped `data/actions.json` directly — of every `creative_capital` effect in the file, exactly two are positive, and both belong to the *same* meeting: `action_1760807005433` ("Employee Effectiveness," cco role), choice `quick_one` (+5, delayed) and choice `take_time` (+9, delayed). Every other `creative_capital` reference in the game (11 occurrences) is a cost (−1 to −3). Round 2's form: `creative_capital_income.anythingOff = "I need to go back and test CC - not clear where CC now builds."` — the designer's own uncertainty about where CC comes from is because there is, structurally, almost nowhere it comes from. #169 already fixed the *discoverability* half (Label Head's Guide "three currencies" topic now names the source) but the underlying economy gap — one meeting, one exec, gates the entire currency's positive supply — is untouched.

**Options (recommendation marked, not decided):**
1. **More CC-granting choices spread across the exec pool**, not concentrated in one cco meeting. Directly fixes the single-point-of-failure; can be authored alongside the A/B/C content pass (a natural byproduct of reworking meetings — some new/reworked choices could grant small CC across multiple execs, not just cco). Ties CC into "your team running things well" rather than one specific dialogue.
2. **CC from milestones** (e.g. a chart placement, a successful breakthrough, a completed tour) as a small passive trickle, mirroring how the leveling spec (plan §6.1) already proposes `xp_per_domain_milestone` for a different currency — reuse the same milestone-hook pattern (`applyChartMilestoneBonuses`, `game-engine.ts:1067`) rather than inventing a new one. Decouples CC income from any single meeting; ties it to actual label performance instead of a dialogue pick.
3. **CC from the CEO lane** — since CEO meetings are being reworked toward "rarer and heavier" (item B), a subset of CEO strategic choices could grant CC as part of that rework, making CC a resource the *player's own* strategic calls generate, not something only an exec meeting can produce. Couples naturally with B's authoring pass — could be decided and authored in the same content-session slice as B.

**Recommendation:** option 1 (spread CC grants across the exec pool) as the most direct fix and the one requiring the least new mechanism — it is pure content authoring, ships in the same session as A/B/C, and doesn't wait on a milestone-hook or CEO-rework dependency the way options 2/3 do. Options 2/3 are good complementary follow-ups (2 especially, since it ties CC to demonstrated label success rather than any single dialogue) but shouldn't gate the first fix.

## (iv) Small-items rider

### H. Buzz hidden at zero during weeks 1–4 building window

Round-2 carryover (first raised round 1, still open per DEVELOPMENT_STATUS.md's 2026-07-12 session log open-threads): a release with no banked hype shows **no Buzz section at all** in its first post-release weeks (1–4, the "building" window) rather than an empty/zero state — so a release that legitimately hasn't built any buzz yet reads as if the Buzz feature doesn't exist for it, rather than "still building." **Candidate fix already recorded:** always show the bar during building weeks, labeled "building," instead of suppressing the section when the value is zero (`client/src/lib/releaseBuzz.ts`, `summarizeReleaseBuzz`). Small, self-contained client-only slice — can ship independent of the content session, but worth folding in if a client agent is in the tree for other reasons.

### I. "Why now" relevance line never noticed by the designer

Round 2's form: `meeting_relevance_whynow.exposure = ["never"]` — despite reactive meetings firing throughout the same playtest (crisis/escalation events were seen and correctly attributed), the designer never registered the specific "why now" relevance line the Tier 2 reactive-meetings work (PRs #133–#145) added to explain *why* a reactive meeting is timely. This is a visibility question, not necessarily a missing-feature one: worth a quick UI-legibility check (is the line rendered somewhere the player's eye actually lands, competing successfully against the urgency-dot and the meeting's own prompt copy?) before assuming it needs content work. Cheap to investigate; flag during the session but don't block content authoring on it.
